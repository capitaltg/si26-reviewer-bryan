# Phase 3: Extraction and Traceability Matrix Design

## Goal

Maintain the worker's requirement extraction and persisted traceability mapping while adding deck-applicability classification so the matrix measures the uploaded oral-proposal artifact instead of every obligation in the procurement. Reviewer judgment, citation verification, report UI, and evaluation tooling remain downstream concerns.

## Scope

The worker reads already-ingested solicitation documents and deck/script page records. A structured Anthropic Bedrock call extracts Section L requirements, Section M factors and weights, SOW/PWS scope statements, presentation limits, incorporated clauses, and amendment or Q&A changes. The call receives the native page text of every solicitation-kind document — base, amendments, Q&A, and attachments — in one pass so supersession across documents can be resolved in one context. It also receives the deck's native text and vision summaries plus any aligned narration, clearly separated as proposal context, so it can identify which solicitation factor the uploaded deck fulfills. Proposal context is used only for applicability classification and is never accepted as an extraction citation.

Extraction is never chunked by document: separate calls could not resolve cross-document supersession or apply one consistent deck scope. The worker measures the complete prompt against a configured input guardrail and fails with an actionable error if it is too large. Each extracted record includes its source document, page number, reference, quoted text, source category, optional weight, an optional supersession pointer, an **applicability classification**, and a short classification rationale.

Only effective requirements enter the matrix. Superseded records remain persisted so the audit trail shows what changed, but a record replaced by a later requirement creates neither a coverage row nor a gap signal.

Coverage mapping applies to obligation-type sources — Section L instructions and SOW/PWS scope statements, including amendment changes to them. An amendment-borne obligation retains its functional `L` or `SOW` source and points to the record it supersedes; `amendment` is reserved for change notes that are not themselves obligations. Section M factors and weights, presentation/formatting limits, incorporated FAR/DFARS clauses, and standalone amendment notes are extracted and persisted for downstream reviewers but are **not** coverage-mapped: a scoring factor or a slide-count limit is not meaningfully `covered`, `partial`, or `missing` against a slide.

Source category alone is too coarse to decide what belongs against the deck. A solicitation's Section L governs the entire multi-phase procurement — written technical factors, a price sheet, cover letters, representations, phase deadlines, and submission mechanics — while the uploaded proposal is only one of those artifacts (for the SA5 fixture, the Factor 3 oral deck). Mapping every Section L obligation against the deck floods the matrix with `missing` rows for obligations that were never the deck's job — procedural deadlines, eBuy submission mechanics, Government-side descriptors, and content that belongs to a written factor, the price sheet, or the cover letter. In one SA5/CTG analysis, roughly 86% of the generated `missing` rows were correctly extracted records that should never have been scored against the deck; this observation motivates the filter but is not treated as a general accuracy measurement.

To separate real deck gaps from cross-artifact noise, extraction classifies every record along three dimensions. The values are required even for source categories that are never coverage-mapped so downstream code can apply one explicit filter:

- **`applies_to`** identifies the proposal scope the record concerns: `deck` for the oral-presentation or demonstration factor and the SOW scope it walks through; `other_component` for a written factor response, price sheet, cover letter, or provision representation; and `administrative` when no proposal artifact addresses the record, such as deadlines, portals, subject-line conventions, award counts, eligibility descriptors, or set-aside policy.
- **`obligation_type`** is `content` for affirmative subject matter an artifact presents or a Government evaluator considers, and `constraint` for a prohibition or rule that must not be violated, such as "no assumptions, caveats, or exceptions." A deck constraint is reviewed for compliance in Phase 4 rather than scored `covered`, `partial`, or `missing`.
- **`obligation_side`** is `quoter` when the record prescribes or describes quoter behavior and `government` when it describes Government behavior or evaluation. A Government-side record can concern the deck — for example, a Section M evaluation factor — but it never becomes a quoter coverage gap.

Each record also carries a non-empty `classification_rationale` that names the inferred factor or explains why the record belongs to another component or to no proposal artifact. These fields are related but not interchangeable: source category remains a separate eligibility gate, so classifying an `M`, `limit`, `FAR`, or standalone `amendment` record as deck-related cannot make it coverage-mappable.

The extractor infers which factor the uploaded deck fulfills — for SA5, the Factor 3 oral presentation — and tags that factor's obligations `applies_to = deck`; there is no upload-time factor selector, and the analysis assumes a single proposal deck. The structured extraction envelope includes a package-level `deck_scope` result with a `resolved` boolean, nullable `factor_ref`, and non-empty rationale. The worker accepts classifications only when `resolved` is true and `factor_ref` identifies one solicitation factor. If the solicitation defines multiple plausible deck artifacts and the supplied proposal context does not identify one unambiguously, extraction fails rather than guessing and silently suppressing possible gaps.

Coverage mapping runs only for records satisfying this complete predicate: the record is effective; its source is `L` or `SOW`; `obligation_side = quoter`; `obligation_type = content`; and `applies_to = deck`. Every other record is persisted with its classification but produces no mapping row. A batched mapping pass states the deck context — native text, vision summaries, and optional aligned narration — once per call and returns, for every selected obligation, one coverage result (`covered`, `partial`, or `missing`) with deck slide references and a non-empty rationale. `covered` and `partial` require at least one existing slide reference; `missing` requires an empty reference list. References are positive, unique slide numbers and are normalized into ascending order before persistence. Selected obligations are split deterministically when their count exceeds the configured maximum per call; unlike extraction, this chunking is safe because mappings have no cross-record dependency. Each call must return exactly one result for every requested requirement id and no others.

The classification is also a downstream contract. Phase 4 loaders and prompts must receive it and must not emit a deck gap for `other_component`, `administrative`, Government-side, or non-content records; deck constraints remain available for compliance review. Phase 5 shows coverage statuses only for the mapped subset and displays effective excluded records without a coverage status in classification-labeled groups: handled by another submission component, administrative/not deck-applicable, and deck constraints or evaluation context. The applicability change is not complete until those downstream readers are updated with the schema rollout; otherwise the matrix can improve while reviewer findings continue to reproduce the same cross-artifact noise.

## Data Model

Extend the existing `requirements` table, which is linked to an analysis and its source solicitation document and already stores source category, reference, quoted text, page number, optional weight, and the optional supersession self-reference. Add three applicability enum columns — `applies_to` (`deck` | `other_component` | `administrative`), `obligation_type` (`content` | `constraint`), and `obligation_side` (`quoter` | `government`) — plus a non-empty text `classification_rationale`. Extraction requires all four classification fields for every new record. Weight remains free text so qualitative Section M phrasing ("significantly more important than price") is preserved rather than forced into a number. The existing `mappings` table remains linked one-to-one with a requirement and stores status, slide references as JSON, and rationale. Because a mapping row exists only for the deck-applicable subset, the requirement-to-mapping relationship is one-to-zero-or-one; database constraints prevent duplicate mappings for a requirement and preserve referential integrity; a mapping is deleted when its requirement is deleted.

The migration leaves all four new columns nullable for legacy rows and adds an all-or-none check: either every classification field is null, or all three enums are non-null and the trimmed rationale is non-empty. It does not invent a default classification. New extraction responses always write the complete non-null set; a legacy null set is treated as unclassified, excluded from mapping and reviewer gap generation, and labeled as such in the report until that analysis is re-extracted. This preserves safety during rollout without classifying old records as `deck` by assumption.

Because a single extraction pass emits every record at once, supersession cannot be expressed as a database id at generation time. The model assigns each record a unique stable key within the pass and expresses supersession by referencing another record's key. Before persistence, the worker validates that every supersession key resolves within the same extraction, no record supersedes itself, and the directed supersession graph is acyclic. It then inserts the records, resolves the keys to database ids, and sets the `supersedes` self-reference. Any invalid key or graph fails the stage.

## Pipeline and Failure Behavior

`run_pipeline` continues to execute `extract` then `map` after ingestion, vision, and optional script alignment and before the existing reviewer and orchestration stages, updating `analyses.stage` and `stage_detail` with human-readable progress. The applicability rollout changes the data and filtering contracts of those existing downstream stages but does not add another LLM stage.

Model responses use the same structured-output mechanism established for the vision pass: a single forced tool call whose arguments the worker reads and validates against a Pydantic schema client-side. The classic Bedrock `InvokeModel` path this deployment is provisioned for rejects `messages.parse()` / `output_config.format` and `strict` tools, so schema enforcement is client-side, not server-side. A `refusal` or `max_tokens` stop reason is treated as untrusted and fails the stage rather than persisting a truncated or empty result.

Extraction citations must refer to an existing solicitation document and page, and the whitespace-normalized, case-folded quoted text must occur on that page. The extractor may choose a shorter contiguous quotation when a requirement spans pages, but it may not persist a paraphrase as quoted source text. Invalid schemas, invalid citations or quotations, or an unresolvable supersession reference fail the current stage and let the existing job-failure handling report the error; no partial matrix is treated as complete.

All solicitation and proposal text is untrusted data. Extraction and mapping prompts delimit it from instructions and tell the model never to follow embedded text that attempts to change its role, tool, schema, or citation rules. Forced tool use and client-side validation remain the enforcement boundary; prompt wording is defense in depth.

Both stages are idempotent under retry. A worker requeue re-runs the pipeline from the start. Extraction validates the complete response first and then, within one database transaction, deletes existing requirements for the analysis — cascading to their mappings — and inserts the replacement set. A failure rolls back the delete and preserves the previous complete set. Mapping likewise validates every batch before atomically replacing mappings for the selected requirements.

## Testing

Unit tests cover structured-response validation, citation and normalized-quotation validation, amendment supersession (including an unresolvable key, self-reference, and cycle), effective-requirement selection, ambiguous deck-scope failure, the oversized full-package failure, and the complete mapping predicate. The mapping cases prove that `M`, `limit`, `FAR`, and standalone `amendment` records are never mapped; only effective, quoter-side, `content`, `applies_to = deck` records from `L` or `SOW` are mapped; excluded records and deck constraints retain a non-empty classification rationale; `covered` and `partial` require valid slide references; and `missing` forbids them. Multi-batch tests prove exact request/response coverage without duplication.

Database-focused tests verify the all-null-or-complete classification constraint, enum and rationale validation, one-to-zero-or-one mapping cardinality, cascade behavior, and atomic replacement on retry, including failure preserving the previous complete set. Downstream regression tests verify that legacy unclassified rows and explicitly excluded records cannot become Phase 4 deck gaps and that Phase 5 groups them without rendering a missing coverage status. A pipeline test verifies the unchanged stage order. Phase 3 deliberately omits ground-truth fixtures, precision/recall reporting, new reviewer calls, finding verification, and new report UI; the required downstream changes are filtering and presentation updates to existing Phase 4/5 behavior.
