# Production environment variables

| Var | Where set | Source |
| --- | --- | --- |
| DATABASE_URL | Vercel + Railway worker | Railway Postgres (public URL for Vercel, internal for worker) |
| AUTH_SECRET | Vercel | `npx auth secret --raw` |
| AUTH_KEYCLOAK_ID / _SECRET / _ISSUER | Vercel | getgaleo.com Keycloak admin (client + realm) |
| AUTH_URL | Vercel | production domain |
| BLOB_READ_WRITE_TOKEN | Vercel (auto) | Blob store attachment |
| POLL_INTERVAL_SECONDS | Railway worker | `2` |
| WORKER_ID | Railway worker (optional) | defaults to hostname |
| BLOB_READ_WRITE_TOKEN | **Railway worker only** | Static token generated from the Vercel dashboard's Storage → store → Tokens tab. Despite sharing a name with the web app's auto-attached `BLOB_READ_WRITE_TOKEN` above, this is a separate credential for a separate service — the worker downloads over plain HTTPS with a static Bearer token; it does not use the web app's OIDC-based Blob access, and the two must never be conflated or shared across services. |
| AWS_ACCESS_KEY_ID / AWS_SECRET_ACCESS_KEY / AWS_REGION | Railway worker | Bedrock IAM credentials + region. Claude is accessed via AWS Bedrock **classic InvokeModel** (`AnthropicBedrock`, model `us.anthropic.claude-opus-4-8` — a US cross-region inference profile) in `worker/src/worker/vision.py` for the per-slide vision pass. Structured output comes from a **forced tool call** (`tools=[record_vision_summary]` + `tool_choice`) — classic Bedrock rejects both `output_config.format`/`messages.parse()` and `strict: true`, so the `summary` schema is enforced client-side. The IAM user needs `bedrock:InvokeModel` (the account's keys are not provisioned for the newer `bedrock-mantle:*` actions), and `AWS_REGION` must be a **US** region for the `us.` profile. Not an Anthropic Console API key. |

> **Never set `AUTH_DEV_LOGIN` in production.** It enables the local-demo
> credentials login (see `web/src/auth.ts`); the code also refuses it when
> `NODE_ENV=production`, but it should not appear in any deployed environment.

## Deploy checklist (Task 10)

- **Postgres pool sizing from Vercel.** Serverless functions can each open
  their own connection pool, so keep it small: set the `pg` Pool `max` to
  1-3 and set `idleTimeoutMillis` so idle connections are released promptly.
  Railway's public Postgres URL also requires `?sslmode=require` on the
  connection string when connecting from Vercel. This is a code change owed
  in `web/src/db/index.ts` at deploy time (not yet applied).
- **Vercel Deployment Protection blocks the Blob webhook.** The
  `onUploadCompleted` callback in `web/src/app/api/upload/route.ts` is
  invoked by Vercel Blob via an unauthenticated webhook request. If
  Deployment Protection is enabled it will block that request, and upload
  rows will never be recorded in the `uploads` table. Either disable
  protection for `/api/upload` or configure a protection bypass token for
  the Blob webhook before relying on uploads in production.
- **Upload constraints.** The upload route caps files at 100MB
  (`maximumSizeInBytes`) and requires the client-requested blob pathname to
  start with `uploads/`; requests that don't meet either constraint are
  rejected with a 400.
